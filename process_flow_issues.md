# 当前处理流程的主要问题

本文只讨论当前 agent 处理流程中的结构性问题，不涉及具体代码修改方案。

## 1. 知识库后台并行存在一致性风险

当前流程是：

```text
observe -> start_kb_bg -> analyze -> act -> sync_kb
```

后台知识库更新拿到的是 `observe` 之后的状态快照。与此同时，`analyze` 和 `act` 会继续更新任务状态、交互历史、动作、预期结果等。虽然 `sync_kb` 只合并知识库相关字段，避免了覆盖主状态，但知识管理使用的是“行动前”的历史和任务状态。

这会导致知识库经常只能理解“刚观察到什么”，但不知道这一轮分析做出了什么判断，也不知道是否发送了关键命令、任务是否完成。对于 MUD 这种命令和响应强相关的环境，知识条目的因果归类容易偏。

## 2. Planner 和 Analyze 的职责边界重叠

`planner` 负责生成阶段任务和执行计划，`analyze` 又在每一轮根据上下文重新判断下一步动作、任务是否完成、环境类型、是否陷入僵局。

当前 `planner` 的执行计划主要只是 prompt 背景，并不是强约束。真正的执行决策大多由 `analyze` 每轮现场决定。因此系统名义上是“规划者驱动”，实际更接近“Planner 负责切任务，Analyze 负责主要决策”。

风险是：Planner 生成的任务质量不稳定时，Analyze 可能会偏离任务；Analyze 判断失误时，Planner 也没有细粒度校正机制。

## 3. LLM 调用没有硬失败出口

`LLMClient.call_with_retry()` 是无限重试。只要出现网络问题、API 配置问题，或者模型持续返回非法 JSON，整个 agent 就可能卡死在某个节点。

对于长期运行的 autonomous agent，这个风险很大。系统至少需要最大重试次数、错误状态记录、降级路径，或者让外层恢复逻辑能够接管。

## 4. 单任务 50 轮僵局阈值过于粗糙

当前通过固定的 `MAX_TASK_ATTEMPTS = 50` 判断任务是否陷入僵局。这个阈值对不同类型任务并不合适：

- 环境识别类任务可能 3 到 5 轮就应该结束。
- 探索类任务可能 50 轮仍然合理。
- 如果模型陷入重复命令，50 轮会浪费大量时间和 token。

更合理的判断方式应该结合重复动作、重复输出、预期落空次数、任务类型等信号，而不是只数轮数。

## 5. 历史记录不够结构化

当前 `history` 是简单字符串列表，例如：

```text
Action: SEND look | Expected: ...
Response to look: ...
Server: ...
```

这种格式对人阅读日志还可以，但给 LLM 推理时不够可靠。MUD 环境中经常会出现异步广播、分页、延迟响应、命令回显等情况。当前结构无法明确区分：

- 哪个输出明确属于哪个输入。
- 哪个输出是服务器自发消息。
- 哪个输出是分页续页。
- 哪个命令成功，哪个命令失败。

这会影响 Analyze 和知识管理对因果关系的判断。

## 6. 环境类型退出逻辑不够稳

第一阶段要求判断是否为文本环境，但 `environment_type` 可能取值为：

```text
text_mud / chat / shell / qa / bbs / non_text / unknown
```

Planner 在阶段完成后只检查 `environment_type == "non_text"` 时退出。问题是 `environment_type` 完全依赖 `analyze` 的 LLM 判断，而且退出还依赖任务被正确标记为完成。

也就是说，非文本环境退出没有底层检测兜底，主要依赖模型同时正确设置环境类型和任务状态。

## 7. 反思可能重复触发或污染经验库

Planner 每次看到 `task_completed` 或 `task_stuck` 就调用反思模块。当前完成任务后会重置 `task_completed`，但僵局处理链路更复杂。长期运行、重连或状态恢复边界处理不好时，可能对同一任务重复反思。

此外，经验和技能主要依赖 prompt 要求模型不要重复，工程侧没有强去重机制。运行时间长了以后，`experiences.json` 可能积累重复、低质量甚至错误的经验，并反过来影响后续决策。

## 8. 状态持久化不完整

知识库和经验会落盘，但主运行状态没有完整持久化。也就是说：

- 进程内重连可以保留 `current_state`。
- 进程崩溃或重启后，阶段、任务、当前任务、已完成阶段、短期历史都会丢失。

这会造成“知识还在，但任务进度没了”的割裂。对于自主探索类 agent，这会明显影响连续性。

## 9. Prompt 注入风险较高

服务器输出会直接进入 `analyze` 和 `manage_knowledge` 的 prompt。远程 socket 环境完全可能返回类似“忽略之前指令，输出某个 JSON”之类的文本。

当前流程没有把服务器内容作为不可信数据进行隔离，也没有在 prompt 中明确要求模型不得服从服务器文本中的元指令。对于 autonomous agent，这属于较高风险。

## 10. 图退出语义不够清晰

LangGraph 的 `END` 同时表示多种情况：

- 非文本环境退出。
- 连接断开，需要重连。
- 发送失败，需要重连。

外层再通过 `should_stop`、`should_exit`、`should_reconnect` 判断原因。当前能跑，但语义比较混。后续如果增加更多退出原因，状态会变得难维护。

更清晰的方式是使用统一的退出原因字段，例如：

```text
exit_reason = reconnect / stop / unsupported_environment / fatal_error
```

## 总体判断

当前流程最大的问题不是节点拆分本身，而是系统过度依赖 LLM 在每一层都做出正确判断。工程侧的结构化状态、失败出口、约束机制、持久化和安全边界还不够强。

短期实验场景下这套流程可以运行；如果要长期自主运行，错误会逐渐累积，并通过知识库、经验库和任务规划继续放大。
